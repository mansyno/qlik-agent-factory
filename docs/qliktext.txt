AI-Assisted Qlik Data Modeling Strategy
Problem Statement

The current 'architect' phase does not always produce correct data models.
One major reason is that imperfections and ambiguity in the source tables make it difficult to automatically infer the intended relationships between tables.

To address this, the modeling process must rely on a structured methodology that combines:

deterministic data profiling
Qlik engine behavior
heuristic modeling rules
AI-assisted reasoning

The goal is to  Design a system capable of automatically generating a correct and maintainable Qlik associative data model by combining:

deterministic data profiling
Qlik engine behavior analysis
heuristic modeling rules
AI-assisted reasoning

The system must prioritize:

1 Analytical correctness
2 Model readability / maintainability
3 Performance
Core Principle of Qlik Modeling

Qlik associates tables based entirely on identical field names.

Therefore:

field naming determines the structure of the associative model

The modeling system must carefully determine:

which fields should share names
which fields must be aliased
which fields must remain isolated
Initial Assumptions

Before the modeling process begins:

tables were curated to belong to the same analytical domain
the dataset was intentionally selected for a single model

Therefore:

tables are expected to contain meaningful analytical relationships
Data Profiling

Before any modeling decisions are made, the system must collect comprehensive metadata about tables and fields.

Whenever possible this information should be gathered using:

Qlik Engine API
Enigma.js
Qlik Desktop engine

The AI assistant should gather as many of the following metrics as computationally feasible.
The list below is extensive but not exhaustive.

Field Classification

Fields must be classified into categories.

Classification should begin with deterministic rules and only use AI reasoning when ambiguity remains.

Field types include:

identifier fields
dimension attributes
measure fields
date/time fields
system / metadata fields
categorical fields
status/flag fields
reference codes

Indicators include:

Identifier Fields
high uniqueness ratio
low null ratio
short numeric or string values
names containing ID / KEY / CODE / NUMBER
Date Fields
date or timestamp datatype
parsable date format
names containing DATE / TIME / CREATED / UPDATED
Measure Fields
numeric values
aggregation behavior
names containing AMOUNT / TOTAL / QTY / VALUE
System / Metadata Fields
ROW_ID
REC_ID
SYS_KEY
LOAD_TIMESTAMP
ETL_BATCH_ID
CREATED_AT
UPDATED_AT

Typical properties:

very high uniqueness
technical naming
no business semantics

These fields usually should not participate in the associative model.

Identifying Table Granularity

Using profiling results, the system must determine the grain of each table.

Grain represents the level at which rows are uniquely identified.

Examples:

Orders → grain: OrderID
OrderLines → grain: OrderID + LineNumber
Inventory → grain: ProductID + WarehouseID + Date

Granularity information is later used to guide:

measure aggregation
master item creation
visualization design
Fact vs Dimension Detection

Tables are classified as:

fact tables
dimension tables

Indicators of fact tables:

large row counts
presence of measures
foreign key patterns
transactional behavior

Indicators of dimension tables:

lower row counts
descriptive attributes
high uniqueness identifiers

AI reasoning may assist in ambiguous cases.

Detecting Candidate Relationships

Relationships between fields are detected using:

field name similarity
value overlap
subset detection
statistical similarity
AI semantic reasoning

Possible cases include:

Case 1 — Natural Matching Fields
same name
same entity
overlapping or subset values

Example:

Orders.OrderID
OrderLines.OrderID

These should remain identical.

Case 2 — Same Name, Different Meaning

Fields share the same name but represent different entities.

Example:

PurchaseOrders.OrderID
CustomerOrders.OrderID

These must be aliased.

Example:

Purchase.OrderID
Customer.OrderID
Case 3 — Same Entity, Different Names

Example:

OrderID
Order_ID
OrderNumber

If profiling confirms equivalence, names should be normalized.

Case 4 — Hidden Relationships

Fields represent the same entity but use unrelated names.

Example:

CustomerID
ClientNumber
AccountCode

Detection relies on:

value overlap
subset detection
statistical similarity
AI reasoning
Relationship Confidence Scoring

Because automatic relationship detection can produce false positives, every candidate relationship must receive a confidence score.

Example scoring inputs:

field name similarity score
value overlap ratio
inclusion ratio
uniqueness compatibility
data type compatibility
semantic similarity
table role compatibility

Example scoring scale:

0.90 – 1.00 → very high confidence
0.70 – 0.89 → likely relationship
0.50 – 0.69 → uncertain
below 0.50 → unlikely relationship

The AI assistant should prefer high-confidence relationships and flag ambiguous cases for further reasoning.

Field Name Normalization

Once relationships are selected, field names must be normalized so that:

fields that should link share the same name
fields that should not link are aliased
Structural Test Load

A minimal load is executed using the normalized names.

Purpose:

observe Qlik engine association behavior
detect synthetic keys
detect circular references

Synthetic keys reveal situations where:

multiple fields connect the same tables

These situations often require link tables or concatenated fact tables.

Multi-Fact Modeling Strategies

When multiple fact tables share dimensions, two strategies exist:

link tables
concatenated fact tables
Link Table Strategy (Default)

A link table contains the shared keys between multiple fact tables.

Advantages:

clear associative structure
separate fact tables preserved
easier model reasoning
better readability

Composite keys may be used inside the link table when needed.

This strategy is preferred unless performance becomes problematic.

Concatenated Fact Tables

Fact tables may be combined into a single table using concatenation.

Advantages:

reduced association complexity
improved performance for very large datasets

Disadvantages:

large sparse tables
more complex field management
When Concatenation Becomes Preferable

Concatenation should be considered when one or more of the following conditions occur:

fact tables exceed tens of millions of rows
link table would contain extremely large composite key cardinality
number of unique key combinations in link table becomes very high
number of shared dimensions between facts becomes large
memory usage becomes excessive
reload time becomes unacceptable
Mandatory Source Table Field in Concatenations

Whenever concatenation is used to union tables, a field identifying the source table must always be added.

Example:

SourceTable
FactType
TransactionType

This rule applies to:

concatenated fact tables
any other union of tables

This allows:

filtering by original source
correct aggregation behavior
clear analytical context
Canonical Date Modeling

Multiple date fields should be unified into a canonical date structure connected to a single master calendar.

Canonical structure:

FactKey
Date
DateType

This allows:

analysis across all dates
analysis by specific date role

Set analysis can be used to filter by specific date types.

Final Model Structure

The final model should approximate a star or snowflake schema.

Typical structure:

fact tables at the center
dimension tables surrounding them
link tables resolving multi-fact relationships
canonical calendar providing time analysis

The resulting model must prioritize:

analytical correctness
clarity of associations
acceptable performance
